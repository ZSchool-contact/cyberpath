לדבר עם שרתים — HTTP ו-curl

פתחנו את הדלת. עכשיו — מה אומרים לעסק שמאחוריה?

בעיר הדיגיטלית שלנו: ה-IP הוא הכתובת, הפורט הוא הדלת. אבל כשנכנסים, צריך לדעת לדבר. היום נלמד את השפה.

Terminal — מה קורה כשנכנסים לאתר
$ curl -v https://example.com
* Connected to example.com port 443
> GET / HTTP/2
> Host: example.com
...
< HTTP/2 200
< server: ECS (nyb/1D18)
< x-powered-by: PHP/7.1.3
<!doctype html><html>...
כל שורה שמתחילה ב-> יצאה מהמחשב שלכם. כל שורה שמתחילה ב-< חזרה מהשרת. הדפדפן עושה בדיוק את זה — רק מסתיר את זה. ה-header x-powered-by: PHP/7.1.3 אומר לתוקף אילו גרסות רצות שם.

ביחידה הזו נבין את השפה הזו — ונתנסה לשלוח בקשות ישירות בעצמנו.

חלק 1 — שפת הרשת

HTTP — שיחה, לא קסם

כשהדפדפן נכנס לאתר הוא לא עושה קסמים. הוא שולח הודעת טקסט לשרת, והשרת עונה בטקסט חזרה.

לקוח
GET /index.html HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0
↓
שרת
HTTP/1.1 200 OK
Content-Type: text/html

<!DOCTYPE html>...
בעיר הדיגיטלית שלנו: ה-IP מכוון לבניין, הפורט פותח את הדלת הנכונה — ו-HTTP הוא מה שאומרים לעסק שמאחורי הדלת. בלי HTTP, גם אם הגעתם לדלת הנכונה — אין שיחה.
למה טקסט? HTTP תוכנן ב-1991 כדי שאנשים יוכלו לקרוא ולכתוב אותו ישירות — בלי כלים מיוחדים. זו הסיבה שאפשר לשלוח בקשות HTTP ישירות מהטרמינל, עם כלי פשוט כמו curl.
קליפ 1

curl -v בטרמינל אמיתי

כל שורה שמתחילה ב-> יצאה מכם. כל שורה שמתחילה ב-< חזרה מהשרת.

חלק 2 — הבקשה

מה יש בבקשה?

לחצו על כל שורה להסבר

HTTP Request
GET /login HTTP/1.1
Method + Path + Version — GET = "תן לי", /login = הדף המבוקש, HTTP/1.1 = גרסת הפרוטוקול. השרת קורא את זה ומבין מה רוצים ממנו.
Host: accounts.google.com
Host header — שרת אחד יכול לארח עשרות אתרים על אותו IP. ה-Host אומר לשרת מי מהם צריך לענות עכשיו. בלי Host — השרת לא יודע לאיזה "עסק" בבניין פונים.
User-Agent: Mozilla/5.0 (Windows NT)
User-Agent — מי שולח את הבקשה: דפדפן, בוט, curl. שרתים יכולים לחסום גישה על בסיס ה-User-Agent. תוקף שרוצה להסתתר ישנה אותו.
Cookie: session=abc123xyz
Cookie — מזהה הסשן שלכם. מי שמצליח לגנוב את ה-Cookie — מתחבר בתור שלכם, בלי שם משתמש וסיסמה. זהו Session Hijacking.
(שורה ריקה — סוף ה-headers)
כל header = מידע. תוקף שמיירט בקשה למד: דפדפן, מערכת הפעלה, עמוד שמבקשים, ולפעמים — Cookie פעיל. זו הסיבה ש-HTTPS חיוני — הוא מצפין את כל זה.
חלק 3 — התגובה

מה השרת מחזיר?

לחצו על כל שורה להסבר — שימו לב במיוחד לאדום

HTTP Response
HTTP/1.1 200 OK
Status Line — גרסת HTTP + קוד תגובה + הסבר. הקוד הוא מה שמעניין: 200 = הצלחה. בהמשך נכיר את כולם.
Content-Type: text/html; charset=UTF-8
Content-Type — מה סוג התוכן שחוזר. text/html לדפים, application/json ל-API. הדפדפן משתמש בזה כדי לדעת איך להציג את הגוף.
Server: Apache/2.4.29 (Ubuntu)
⚠ Server header — השרת חושף את שמו וגרסתו. Apache 2.4.29 = גרסה ספציפית = CVEs ידועים. תוקף יחפש חולשות עבורה. שרת מאובטח מסיר את ה-header הזה.
X-Powered-By: PHP/7.1.3
✕ X-Powered-By — חשיפה גדולה. PHP 7.1.3 שיצאה ב-2017 — לא מקבלת עדכוני אבטחה מ-2019. תוקף שמוצא זאת יחפש exploits ל-PHP 7.1. אין שום סיבה להשאיר header זה.
(שורה ריקה — תחילת גוף התגובה)
<!DOCTYPE html><html>...
קודי תגובה

מה אמר השרת?

לחצו על כל קוד — כולל הזווית הביטחונית

200
OK
הבקשה הצליחה
›
השרת מצא את מה שביקשתם. מנקודת מבט סייבר: 200 = פורט פתוח + שירות פעיל. כלי סריקה מחפשים 200 כדי לזהות נקודות כניסה.
301
Moved Permanently
הועבר לכתובת חדשה
›
השרת מחזיר Location: עם הכתובת החדשה. ככה http:// מפנה אתכם ל-https:// אוטומטית. דפדפן עוקב לבד, curl דורש -L.
403
Forbidden
גישה נדחתה
›
השרת הבין את הבקשה — וסירב. חשוב: 403 מאשר שהמשאב קיים. תוקף שמקבל 403 על /admin יודע שיש שם עמוד — רק שחסום. 404 היה עדיף מביטחוני.
404
Not Found
הדף לא קיים
›
הדף לא קיים בשרת. כלי סריקה מנסים אלפי נתיבים (/admin, /.env, /backup.zip) ומחפשים 200 במקום 404. אפשר ללמוד הרבה על מבנה האתר.
500
Internal Server Error
הקוד קרס
›
קוד הצד השרת קרס. לפעמים חושף stack trace עם שמות קבצים, שורות קוד, גרסאות ספריות. מידע יקר לתוקף שרוצה להבין את הארכיטקטורה הפנימית.

קליפ 2

DevTools לעומת curl

הדפדפן ו-curl עושים בדיוק אותו דבר. הדפדפן מסתיר — curl מראה הכל.

מתחת לפני השטח

curl -v — לראות את השיחה המלאה

הפלג -v חושף הכל. לחצו על כל קטע.

curl -v https://example.com
* Connected to example.com port 443
חיבור + TLS — לפני שנשלחת מילה, curl מצא את ה-IP, התחבר לפורט 443, ועשה לחיצת יד TLS שמצפינה את כל מה שיבוא אחריה.
> GET / HTTP/2
> Host: example.com
> user-agent: curl/8.1.2
הבקשה שלנו (>) — כל שורה שמתחילה ב-> יצאה מהמחשב שלנו. זו בדיוק הבקשה שדפדפן שולח בכל ניווט לאתר.
< HTTP/2 200
< content-type: text/html
< server: ECS (nyb/1D18)
התגובה מהשרת (<) — 200 OK, השרת מזהה את עצמו. כל מה שמתחיל ב-< הגיע מהשרת. כאן רואים את ה-headers שדפדפן מסתיר מכם.
<!doctype html><html>...
גוף התגובה — ה-HTML שהדפדפן מפרש ומציג. curl מדפיס אותו גולמי. זה בדיוק מה שהדפדפן מקבל לפני שהוא "מצייר" אותו.
זווית ביטחונית

מה שרתים מגלים על עצמם

curl -I חושף לעיתים Server + X-Powered-By. גרסה ספציפית = CVEs ידועים.

דוגמה אמיתית: שרת שמחזיר X-Powered-By: PHP/5.6.40 — PHP 5.6 הגיעה לסוף חיים ב-2018. תוקף שמצא זאת יחפש CVEs ל-PHP 5.6 ויתחיל לנצל אותם. כל זה מ-header אחד.

סיכום

מה למדנו?

HTTP הוא שיחת טקסט. לקוח שולח method + path + headers. שרת עונה עם קוד + headers + גוף. הדפדפן עושה בדיוק את זה — curl פשוט מראה לנו.
Headers = מידע שחושפים. Server, X-Powered-By, גרסאות — כל אלה עוזרים לתוקף לזהות חולשות. שרת מאובטח מסיר אותם.
curl הוא כלי יומיומי. GET, -I לheaders, -v לשיחה מלאה. מפתחים, מנהלי שרתים, חוקרי אבטחה — כולם משתמשים בו.
